# SDD - KnowledgeGame

## Status
Under construction and iteratively refined through discovery questions.

---

## 1. Overview

KnowledgeGame is an adaptive learning app driven by questions. A user enters any topic, and an LLM dynamically generates sub-items and questions for that topic. The core goal is to detect what the user knows versus what they do not know, then personalize learning continuously.

---

## 2. Screen Flow

### Page 1 - Search
- Minimal interface inspired by Google-style search.
- Central text input where the user types a topic, for example: AZ-900 or automotive mechanics.
- On submit, the user is routed to Page 2.

### Page 2 - Learning Session
- Left dashboard: topic structure and user proficiency per sub-item.
- Right panel: sub-item overview.
- Center panel: one question at a time.

### Supported Question Types
- Multiple choice
- Single choice
- Fill in the blank
- True/False
- Timed and untimed variants

### Core Mechanic - Conveyor Belt
Learning is continuous. There are no hard session endings.

- Only one visible question at a time.
- The user can stop and resume anytime.
- A horizontal queue previews upcoming questions without revealing answers.

### Queue Settings

| Option | Label | Description | Default |
|---|---|---|---|
| 1 | Queue Depth | Max number of ready questions kept in memory | 5 |
| 2 | Refill Trigger | Generate more when queue reaches this size | 2 |

Queue behavior:
- Keep up to Queue Depth questions available.
- Trigger generation when remaining queue count reaches Refill Trigger.
- Show a small fallback loading state only if queue reaches zero.

## 3. Adaptive Algorithm

Architecture uses a hybrid approach: Bayesian-style signals plus spaced repetition heuristics.

Per sub-item, track:
- Historical accuracy
- Response speed
- Error patterns
- Time since last exposure

Prompt instructions to the LLM should request:
- Targeted remediation for weak concepts
- Progressive difficulty for mastered concepts
- Intelligent review scheduling for stale concepts

---

## 4. Domain Model

```text
Topic
     -> Item (1..N)
          -> SubItem (1..N)
               -> Question (1..N)
```

Entities:
- Topic: user-entered subject
- Item: major section generated by LLM
- SubItem: granular concept inside an item
- Question: generated assessment unit linked to a sub-item

Mute rules:
- Muting a topic excludes all its items/sub-items from generation.
- Muting an item excludes all its sub-items.
- Muting a sub-item excludes only that sub-item.

Muted entities remain visible but are ignored by generation logic.

---

## 5. Content Generation

- Engine: LLM API (for example Claude or OpenAI)
- Dynamically generated: sub-items, questions, options, correct answer, explanation
- Progressive generation strategy to keep UX responsive
- Streaming responses are recommended to reduce perceived latency

---

## 6. Persistence and Auth

Current POC scope:
- No login required
- Session progress can be stored locally
- Generated questions should also be persisted for reuse

Recommended persistence stack:
- Database: PostgreSQL (for example Supabase)
- ORM: Prisma

Reuse strategy:
- Persist every generated question with metadata.
- Before calling the LLM, check if enough reusable questions already exist.
- Reuse first, generate only when required.

- Styling: Tailwind CSS

## 8. Gamification

Progress features:
- XP per correct answer
- Streak tracking
- Lives/hearts logic
- Historical progress bars per sub-item
- Proficiency ladder: Beginner -> Basic -> Intermediate -> Advanced -> Expert

Visual feedback:
- Immediate correct/incorrect feedback
- Explanation shown after answer
- Celebration moments for milestones

Spaced repetition:
- Increase review interval after correct answers
- Decrease/reset interval after wrong answers
- Prioritize overdue review questions

---

## 10. Transitions and Loading

Page 1 -> Page 2 should feel memorable:
- Intelligent animated transition
- Dynamic rotating status copy while generating
- Streaming reveal of sub-items and first questions

UX principles:
- Avoid static spinners when possible
- Prefer skeleton states and purposeful motion
- Keep interactions responsive and informative

---

## 11. Difficulty Progression

- Always start at level 1.
- Increase difficulty only after consistent correctness.
- Allow drops in difficulty after mistakes.
- Pass current difficulty to generation prompt.

---

## 12. Non-Functional Requirements

- All UI and messages must be in English.
- Fast perceived performance is a priority.
- Progressive loading should avoid hard blocking.
- Strong visual feedback for every interaction.
- Colorblind-friendly communication patterns.
